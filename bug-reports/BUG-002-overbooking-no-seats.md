# BUG-002 — Перепродажа мест: бронь проходит при отсутствии мест

| Поле | Значение |
|------|----------|
| ID | BUG-002 |
| Severity | Major |
| Priority | P1 |
| Компонент | Бронирование |
| Окружение | staging, API v2.3 |
| Статус | Open |

## Описание
Когда у тура заканчиваются места, UI блокирует кнопку, но **прямой запрос** к API
всё равно создаёт заявку — возникает перепродажа (overbooking).

## Шаги воспроизведения
1. Выбрать тур, где `seats_total` уже исчерпан (например, бутик-тур на 2 места, 2 уже проданы).
2. Отправить `POST /bookings` напрямую с `pax = 1`.
3. Проверить заявки и доступность мест.

## Фактический результат
Заявка создаётся (201), суммарно продано больше мест, чем `seats_total`.

```sql
SELECT t.id, t.seats_total, SUM(b.pax) AS sold_pax
FROM tours t
JOIN bookings b ON b.tour_id = t.id AND b.status IN ('new','paid')
WHERE t.id = 2
GROUP BY t.id, t.seats_total
HAVING SUM(b.pax) > t.seats_total;
-- seats_total = 2, sold_pax = 3
```

## Ожидаемый результат
API отклоняет бронь при отсутствии мест (409/422), заявка не создаётся.
Проверка вместимости — на сервере, не только в UI.

## Вложения
- Postman-запрос с прямым `POST /bookings`.
- SQL-выборка с превышением вместимости.
