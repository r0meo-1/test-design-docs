# BUG-001 — Двойное списание при повторной отправке оплаты

| Поле | Значение |
|------|----------|
| ID | BUG-001 |
| Severity | Critical |
| Priority | P1 |
| Компонент | Оплата / платёжный шлюз |
| Окружение | staging, Chrome 124, API v2.3 |
| Статус | Closed (Fixed) |

## Описание
При повторной отправке запроса оплаты (двойной клик / повтор по таймауту) создаётся
**два успешных платежа** на одну заявку — клиент списывается дважды.

## Шаги воспроизведения
1. Создать заявку, перейти к оплате (сумма 170 000 ₽).
2. Нажать «Оплатить» и сразу повторить запрос (двойной клик или ретрай при таймауте).
3. Открыть заявку и платежи в CRM.

## Фактический результат
Два платежа `succeeded` по 170 000 ₽ на одну заявку. SQL-проверка суммы показывает расхождение:

```sql
SELECT b.id, b.amount, SUM(p.amount) AS paid_sum
FROM bookings b
JOIN payments p ON p.booking_id = b.id AND p.status = 'succeeded'
WHERE b.id = 1024
GROUP BY b.id, b.amount;
-- amount = 170000, paid_sum = 340000
```

## Ожидаемый результат
Повторный запрос с тем же ключом идемпотентности **не** создаёт второй платёж;
по заявке ровно один успешный платёж, равный её сумме.

## Вложения
- Лог запросов (Charles): два POST `/payments` без проверки idempotency-key.
- Скриншот списка платежей в CRM.

## Примечание
Связанный авто-кейс: `qa-api-tests` → «Idempotency». Регресс закрыт после добавления
обязательного idempotency-key на стороне API.
