# Тест-план — система онлайн-бронирования туров

| Параметр | Значение |
|----------|----------|
| Продукт | Web-приложение турагентства: каталог, бронирование, оплата, личный кабинет |
| Версия | 2.x (релизы раз в 1–2 недели) |
| Автор | Роман Неклюдов, QA Engineer |
| Окружения | dev → staging → production |

## 1. Цель

Подтвердить, что бизнес-критичные сценарии (бронирование, оплата, отмена, возврат, перерасчёт)
работают корректно, интеграции с платёжным шлюзом и поставщиками турпродукта стабильны,
а данные в CRM остаются целостными.

## 2. Объём тестирования

**В объёме:**
- Каталог и карточка тура (поиск, фильтры, цена, наличие мест).
- Сценарий бронирования (выбор тура, количество туристов, контактные данные).
- Оплата и платёжные статусы (успех, отказ, таймаут, повтор).
- Отмена, возврат, перерасчёт стоимости.
- Личный кабинет: история заявок, статусы.
- REST API интеграций (см. репозиторий `qa-api-tests`).
- Целостность данных в CRM (см. репозиторий `sql-data-quality`).

**Вне объёма:**
- Нагрузочное тестирование (отдельный план).
- Внутренняя бухгалтерия поставщиков.

## 3. Виды тестирования

Функциональное · регрессионное · smoke · интеграционное · приёмочное · кроссбраузерное · API · базовые проверки безопасности (авторизация, доступ к чужим заявкам).

## 4. Подход

- Тест-дизайн: классы эквивалентности, граничные значения, таблицы решений, попарное тестирование.
- Каждая фича сопровождается чек-листом и набором тест-кейсов (`test-cases/`).
- Перед релизом — smoke на staging; после релиза — smoke на production (`checklists/`).
- Дефекты заводятся по шаблону (`bug-reports/`) с приоритизацией.
- Прослеживаемость требований — `traceability-matrix.md`.

## 5. Критерии входа

- Сборка задеплоена на staging и доступна.
- Известные блокеры прошлой итерации закрыты или приняты как риск.
- Тестовые данные и доступы подготовлены.

## 6. Критерии выхода

- 100% тест-кейсов P1/P2 выполнены.
- Нет открытых дефектов severity Blocker/Critical.
- Smoke на production пройден после деплоя.
- Известные дефекты задокументированы и согласованы с командой.

## 7. Среда и инструменты

| Категория | Инструменты |
|-----------|-------------|
| API | Postman, Swagger, Newman |
| Трафик/отладка | Chrome DevTools, Charles Proxy |
| Данные | SQL, PostgreSQL |
| Трекинг | Jira, YouTrack, Confluence |
| Автоматизация | GitHub Actions, n8n, Telegram Bot API |

## 8. Риски

| Риск | Митигация |
|------|-----------|
| Платёжный шлюз нестабилен на стороне провайдера | мониторинг статусов, проверки идемпотентности и повторов |
| Рассинхронизация данных CRM ↔ оплаты | регулярные SQL data-quality проверки |
| Короткое окно релиза | приоритезация smoke- и регресс-набора |
